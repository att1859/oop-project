## 1. Motivation
- 왜 하스스톤식 카드 배틀을 선택했는가?
    - 카드에서 하수인, 마법, 필드, 덱 등을 객체로 정의하기 적절하다고 생각했고, 카드 추가 등의 상황으로 추후 유지보수 상황이 자연스럽기 때문에 선택했다.
- 게임 개발 관심과 어떤 관련이 있는가?
    - 실제 게임에서도 무수히 많은 캐릭터, 맵, 상호작용이 있고, 대표적인 카드게임인 하스스톤을 모티브로 만들면 도움이 될 것이라 생각했다.

## 2. OOP Design
- Card, Player, Deck, Hand, Field, GameManager 역할
    - Card: 플레어이가 낼 수 있는 모든 카드들의 공통 부모 클래스. 이름, 비용, 설명을 가지며, play() 메서드를 통해 효과를 실행한다.
    - Player: 체력, 마나, 덱, 핸드, 필드 상태를 관리하는 객체. 카드를 뽑고, 카드를 사용하며, 피해를 받는 동작을 수행한다.
    - Deck: 플레이어가 사용할 카드 목록을 관리하고, 카드를 섞고 뽑는 기능을 담당한다.
    - Hand: 플레이어가 현재 들고 있는 카드 목록을 관리한다.
    - Field: 소환된 하수인들을 관리한다.
    - GameManager: 턴 진행, 마나 회복, 카드 사용 처리, 승패 판정, 전투 로그 기록 등 전체 게임 흐름을 제어하는 관리자 객체이다.
- 왜 UI와 game logic을 분리했는가?
    - 추후 pygame 혹은 webui 등 ui 부분을 업그레이드 하기 용이하도록 ui 적용 또한 oop 디자인을 따랐다.
## 3. Before / After
- 절차지향으로 만들었을 때 문제
    - 우선, 추후 유지 보수 시 여러 부분에서 코드 수정이 필요할 수 있다.
    - 각각의 기능들을 분리하여 보기 어렵고, 오류 발생 시 추적이 힘들 수 있다.
- OOP로 만들었을 때 장점
    - 추후 유지 보수 시 용이하다. 예를 들자면 새로운 카드 추가 시 큰 흐름의 변경 없이 Card를 상속한 새로운 클래스를 추가하는 방식으로 할 수 있다.
    - 데이터 보호 및 안정성: 캡슐화를 통해 객체의 내부 상태 변경 방식을 통제할 수 있다. 예를 들어 HP나 마나를 직접 수정하기보다 take_damage(), spend_mana() 같은 메서드를 통해 변경하면 예상치 못한 오류를 줄일 수 있다.
    - 각각의 기능에 대한 이해가 모듈마다 적용돼어 readability가 올라간다.
